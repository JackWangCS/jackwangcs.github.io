背景

在Flink 1.9之前，面向用户的Table API & SQL虽然是统一的，但是针对Stream和Batch任务分别采用了独立的API：DataStream API和 DataSet API。两套API的相对独立，在翻译层具有不同的代码路径，并且在Runtime层也要根据不同的API构造执行计划JobGraph，使得整个架构能够复用的模块有限，不易拓展：

在Flink 1.9时，引入了来自阿里贡献的Blink Planner。Blink Planner将批处理作为流处理的一个特列（Bounded Stream），尽量对通用的处理和优化逻辑进行抽象和复用，通过 Flink 内部的 Stream Transformation API 实现流 & 批的统一处理，替代原 Flink Planner 将流 & 批区分处理的方式。新架构通过灵活的插件化方式来兼容旧版本的Planner，从而可以逐渐演化架构。

在Flink 1.11时，Flink将默认的Planner切换到了Blink Planner，最终将逐步淘汰旧版本的Planner：

Flink SQL执行流程

High level的流程与Spark SQL等类似，包含：

  • 将 SQL文本 / TableAPI 代码转化为逻辑执行计划（Logical Plan）
  • Logical Plan 通过优化器优化后，转化为物理执行计划（Physical Plan）
  • 通过Code-Gen技术生成 Transformations ，进一步编译为可执行的 JobGraph 提交运行

具体地，对于Blink Planner，详细的步骤如下：

  1. Parsing：语法解析，把 SQL 语句转换成为一个抽象语法树（AST），在 Calcite 中用 SqlNode 来表示
  2. Validation：语法校验，根据CatalogManager和FunctionManager中的元数据信息进行验证，例如查询的表、使用的函数是否存在等，校验以后产生由Operation构成的逻辑执行计划（Operation DAG）
  3. RelNode Generation：将Operation DAG转换为RelNode(Relational expression) DAG
  4. Optimization：优化器利用各种的优化规则和统计信息，对RelNode DAG进行等价变换，优化执行计划。目前，流&批处理共享绝大多数的优化规则，但是批处理没有Stage相关的概念，而流处理则不支持排序等。因此，Blink Planner定义了两个独立的规则集和Physical Rels：BatchPhysicalRel和StreamPhysicalRel。
  5. Transformation：Blink将Physical Rel DAG转化为ExecNode。在ExecNode中，将会执行CodeGen和non-code的operators。最终，ExecNode DAG将会被转化为Transformation DAG
  6. Generation of an executable JobGraph：Transformation将会被StreamGraphGenerator转化为StreamGraph、并最终转换为可执行的JobGraph。

TableEnvironment

TableEnvironment是用于创建Table&SQL程序的上下文执行环境，是程序的入口。主要功能包含：

  • 连接外部系统
  • 表及元数据的注册和检索，依赖于Catalog系统
  • 执行SQL语句
  • 提供详细的配置选项

TableEnvironment接口是面向纯Table程序，如果需要从Flink DataStream和Flink DataSet进行转换，那么需要使用语言相关的各种TableEnvironment实现进行桥接。

在Flink 1.8以及之前的版本，有多达7个TableEnvironment。从Flink 1.9开始，根据FLIP-32，对Flink的Table&SQL部分进行了重构和优化，目前保留了5个TableEnvironment，分别是：

  • org/apache/flink/table/api/TableEnvironment.java：顶层接口，针对不同JVM语言和流批数据的统一。但是，TableEnvironment 由于Java和Scala的类型推导还未统一，所以目前还不支持注册 UDTF 和 UDAF，。
  • java/BatchTableEnvironment.java和scala/BatchTableEnvironment.scala：相比 TableEnvironment，BatchTableEnvironment 提供了 DataSet 和 Table 之间相互转换的接口，如果用户的程序需要额外使用到 DataSet API，则需要使用 BatchTableEnvironment。上文提到，DataSet API将会被Deprecated掉
  • java/StreamTableEnvironment.java和scala/StreamTableEnvironment.scala：相比 TableEnvironment，StreamTableEnvironment 提供了 DataStream 和 Table 之间相互转换的接口，如果用户的程序需要额外使用到 DataStream API，则需要使用 StreamTableEnvironment。注意，StreamTableEnvironment统一使用Unified DataStream，支持以Unbounded/Bounded Stream的方式执行Stream和Batch作业。

EnvironmentSettings

EnvironmentSettings包含了用于初始化TableEnvironment的各种参数，包括：

  • Planner Factory Class，默认为BLINK_PLANNER_FACTORY（”org.apache.flink.table.planner.delegation.BlinkPlannerFactory”）
  • Executor Factory Class，默认为BLINK_EXECUTOR_FACTORY（“org.apache.flink.table.planner.delegation.BlinkExecutorFactory”）
  • Built-in Catalog Class and Built-in Database Class
  • Is Streaming Mode，默认为true

EnvironmentSettings的Builder用于客户代码构造EnvironmentSettings。

TableEnvironmentImpl

TableEnvironmentImpl是TableEnvironmentInternal的实现：

TableEnvironmentInternal继承自TableEnvironment是内部接口，主要用于兼容Legacy Planner中的TabelImpl实现，之后会将其中定义的方法移动到TableEnvironmentImpl。

在TableEnvironmentImpl的create()方法中，根据EnvironmentSettings的配置，通过SPI的方式创建具体的Executor和Planner实现。

executeSql(String)用于执行一个Sql语句：

在通过Parser解析得到Operation以后，通过executeOperation(Operation)执行该Operation。

Executor & Pipeline

Executor用于执行由Planner生成的由Transformation表示的DAG，从而将TableEnvironment与运行时解耦。Executor首先将Transformation DAG转换为Pipeline（由StreamGraph表示，StreamGraph是Pipeline的唯一子类），之后委托对应的TableEnvironment进行执行：
