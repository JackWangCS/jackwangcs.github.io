Overview of Classloading in Flink

Three groups:

  1. The Java classpath: it includes the JDK libraries, and all code in Flink’s /lib folder.
  2. The Flink Plugin Components: The plugins code in folders under Flink’s /plugins folder. Flink’s plugin mechanism will dynamically load them once during startup.
  3. The Dynamic User Code: These are all classes that are included in the JAR files of dynamically submitted jobs, (via REST, CLI, web UI). They are loaded (and unloaded) dynamically per job.

YARN

YARN classloading differs between single job deployments and sessions:

  • When submitting a Flink job directly to YARN (via bin/flink run -t yarn-per-job ...), dedicated TaskManagers and JobManagers are started for that job. Those JVMs have user code classes in the Java classpath. That means that there is no dynamic classloading involved in that case for the job.
  • When starting a YARN session, the JobManagers and TaskManagers are started with the Flink framework classes in the classpath. The classes from all jobs that are submitted against the session are loaded dynamically.
  • Application mode is a bit different. Users jar are loaded two twice: 1. it’s loaded by Java classpath. 2. it’s loaded by FlinkUserCodeClassLoader dynamically

Classloaders

Two classloaders:

  • Java’s application classloader, which has all classes in the classpath
  • The dynamic plugin/user code classloader

By default, Flink inverts classloading order, meaning it looks into the dynamic classloader first, and only looks into the parent (application classloader) if the class is not part of the dynamically loaded code.

Please note that certain classes are always resolved in a parent-first way (through the parent ClassLoader first), because they are shared between Flink’s core and the plugin/user code or the plugin/user-code facing APIs. The packages for these classes are configured via classloader.parent-first-patterns-default and classloader.parent-first-patterns-additional. To add new packages to be parent-first loaded, please set the classloader.parent-first-patterns-additional config option.

  1. classloader.parent-first-patterns-default: java.;scala.;org.apache.flink.;com.esotericsoftware.kryo;org.apache.hadoop.;javax.annotation.;org.slf4j;org.apache.log4j;org.apache.logging;org.apache.commons.logging;ch.qos.logback;org.xml;javax.xml;org.apache.xerces;org.w3c
  2. classloader.parent-first-patterns-additional: (default “”)

ChildFirstClassLoader & ParentFirstClassLoader

此外，Flink实现了根据类的解析顺序，定义了：

  • ParentFirstClassLoader：Java的ClassLoader的默认行为，采用双亲委托的类加载方式，即优先从父ClassLoader中加载类，如果父ClassLoader中没有该类，再从子ClassLoader中加载。

  • ChildFirstClassLoader：优先从ChildFirstClassLoader加载类，如果加载失败，才尝试从父ClassLoader中加载类。为了避免ChildFirstClassLoader从用户的Jar文件和库中加载系统类和Flink的核心类，ChildFirstClassLoader还支持提供一组类的前缀匹配模式，匹配模式的类总是优先从父ClassLoader中加载。

两者的父类都为FlinkUserCodeClassLoader:

FlinkUserCodeClassLoader在URLClassLoader支持设置传入一个classLoadingExceptionHandler，用于在加载失败做一些额外的处理。
