资源调度器是YARN最核心的组件之一，是一个插拔式的服务组件，负责整个集群资源的管理和分配。YARN提供了三种资源调度器实现：

  • FIFO
  • Capacity Scheduler
  • Fair Scheduler

在YARN中，资源调度器以层级队列方式组织资源，符合公司的树形组织架构，有利于资源在不同组织间的资源分配和共享，从而提高集群资源利用率。

资源调度架构

资源表示模型

YARN支持内存和CPU两种资源类型的管理和分配，采用了动态资源分配机制。NodeManager启动时，会向ResourceManager进行注册，注册信息中包含：

  • yarn.nodemanager.resource.memory-mb，可分配的物理内存总量
  • yarn.nodemanager.vmem-pmem-ratio，任务使用物理内存与虚拟内存的比例，默认为2.1
  • yarn.nodemanager.resource.cpu-vcores，可分配的虚拟CPU个数，默认为8。YARN允许管理员根据将每个物理CPU划分成若干v-cores
  • 磁盘容量、网络和磁盘I/O等

YARN通过调度语义来决定能够给用户带来的服务承诺。支持的调度语义：

  • 请求某个特定节点上的特定资源量
  • 请求某个特定机架上的特定资源
  • 将某些节点加入或移除黑名单，不再为该客户分配这些节点上的资源
  • 请求归还某些资源

但不支持：超细粒度资源、动态调整Container资源、请求任意节点或者任意机架的特定资源量。

资源调度模型

双层资源调度模型

YARN采用了双层资源调度模型，第一层是ResourceManager中的ResourceScheduler将资源分配各个ApplicationMaster，第二层是ApplicationMaster将资源分配给内部的各个任务。

YARN采用异步资源分配，当ResourceScheduler将资源分配给一个应用程序后，并不会立即下发到对应的ApplicationMaster，而是暂存到缓冲区中等待ApplicationMaster通过周期性心跳主动获取，采用了pull-based通信模型。

资源保存机制

资源保证分为：增量资源分配（Incremental placement）和一次性资源分配（all-or-nothing），两种机制各有优缺点：对于增量资源分配机制，资源预留会导致资源浪费，从而降低资源的利用率；而一次性资源分配机制可能会产生饥饿现象。YARN采用了增量资源分配机制。

YARN层级队列管理机制

层级队列组织方式具有的特点：

  • 子队列
    • 队列可以嵌套，每个队列均可以包含子队列
    • 用于只能将应用程序提交到叶子队列
  • 最少容量（capacity）
    • 每个子队列都有一个最少容量，表示可以使用父队列的容量的百分比
    • 调度器总是优先选择当前资源使用率最低的队列用于分配资源。
    • 最少容量不是总会保证的容量
    • 最少容量大于0，但小于最大容量
  • 最大容量（maximum-capacity）
    • 为了防止队列超量使用资源而设置的最大容量，是资源使用的上限
    • 默认情况下，最大容量无限大

资源抢占模型

在资源调度器中，每个队列可设置一个最小资源量和最大资源量。为了提高资源利用率，资源调度器会将负载较轻的队列的资源暂时分配给负载重的资源，但当负载较轻的队列突然收到新提交的应用程序时，需要调度器去抢占（为了防止资源浪费，会首先等待其他队列释放资源，超过等待时间则进行抢占）本该属于该队列的资源进行分配。

ResourceManager在参数yarn.resourcemanager.sheduler.monitor.enable设置为true，并且资源调度器实现了PreemptableResourceScheduler接口时才会启用资源抢占功能。资源抢占策略通过可插拔的实现SchedulingEditPolicy接口的策略类确定，YARN默认为CapacityScheduler提供了ProportionalCapacityPreemptionPolicy，可以通过yarn.resourcemanager.scheduler.monitor.policies指定。ResourceManager使用了SchedulingMonitor对这些策略进行封装，SchedulingMonitor会周期性地调用策略类中的editSchedule()函数来决定需要抢占的Container资源。整个过程如下：

ProportionalCapacityPreemptionPolicy的主要算法分为三步：

  1. 计算理想的资源分配情况idealAssigned
  2. 基于idealAssigned选择应该将要被抢占的资源，并加入到toPreempt
  3. 遍历toPreempt中的资源，根据preempted计算等待时间，决定是等待还是直接发送kill_container命令
