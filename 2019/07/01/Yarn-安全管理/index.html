由于Hadoop集群都部署在具有防火墙保护的局域网内且只允许企业内部人员访问，因此Hadoop添加安全机制的动机并不是防御外部黑客的攻击，而是为了更好地让多个用户在共享Hadoop集群环境下安全高效地使用集群资源。由两部分组成认证（authentication）和授权（authorization）组成，Hadoop2.0中的认证机制采用了Kerberos和Token的两种方案，授权则是通过引入访问控制列表（ACL）实现。

认证机制

由于Kerberos在高并发情况下效率较低，因此Hadoop采用了Kerberos和Token两种结合的认证方式：客户端与服务器首先通过Kerberos进行认证，然后客户端从服务器端获得一个Token。之后，客户端访问服务器资源时，都通过该Token进行认证，即基于授权令牌（Delegation Token）的认证机制。

客户端与NameNode，ResourceManager之间的初次通信采用Kerberos进行认证，之后便使用Delegation Token的方式减少开销。但是DataNode与NameNode，NodeManager与ResourceManager之间始终采用Kerberos认证机制。

Hadoop中Token类：

当一个客户端或服务使用Token向另外一个服务进行认证时，步骤如下：

  1. 客户端将TokenIdentifier发送给服务器端

  2. 服务端使用TokenIdentifier和masterKey（密钥）计算TokenAuthenticator和Token。masterKey是客户端经过Kerberos认证后获取的，并周期性地心跳获取最新的masterKey。

    TokenAuthenticator=HMAC-SHA1(masterKey, TokenIdentifier)

  3. 服务器端检查新的Token是否合法，一个Token合法，当且仅当Token在内存中存在，并且当前时间仍在有效期内。

  4. 如果Token合法，客户端和服务端分别将TokenAuthenticator作为密钥、DIGEST-MD5作为认证协议进行双向认证。

YARN中各类Token

  • ResourceManager Delegation Token，该Token用于应用程序与ResourceManager进行交互，由RMDelegationTokenSecretManager管理和维护。
  • YARN Application Token，用于保证ApplicationMaster与ResourceManager之间的通。该Token的masterKey由RM传递给NM，并保存到ApplicationMaster Container的私有目录下。当NM启动ApplicationMaster时，所有的私有Token将会加载到ApplicationMaster的UGI中。该Token由RM中的AMRMTokenSecretManager管理和维护。
  • YARN NodeManager Token，ApplicationMaster与NodeManager通信时使用，由RM中的NMTokenSecretManagerInRM管理和维护。
  • YARN Container Token，ApplicationMaster与NodeManager通信启动Container时需要使用Container Token。ApplicationMaster通过ContainerManagementProtocol#startContainer与NodeManager进行通信时使用，由RM中的RMContainerTokenSecretManager管理和维护。
  • YARN Localizer Token，用于保证ContainerLocalizer与NodeManager之间的通信安全。ContainerLocalizer负责在任务运行之前从HDFS下载各类所需的文件资源，以构建一个本地的执行环境。在文件下载过程中，ContainerLocalizer通过RPC协议LocalizationProtocol不断向NodeManager汇报信息。
  • MapReduce Client Token，用于保证MapReduce JobClient与MapReduce ApplicationMaster之间的通信安全。包括MapReduce Job Token和MapReduce Shuffle Secret。
