Introduction

Project Tungsten focuses on substantially improving the efficiency of memory and CPU for Spark to push performance closer to the limits of modern hardware.

Tungsten includes parts:

  • Memory Management and Binary Processing: leveraging application semantics to manage memory explicitly and eliminate the overhead of JVM object model and garbage collection
  • Cache-aware computation: algorithms and data structures to exploit memory hierarchy
  • Code generation: using code generation to exploit modern compilers and CPUs
  • No virtual function dispatches: reducing multiple CPU calls which can have a profound impact on performance when dispatching billions of times
  • Intermidiate data in memory vs CPU registers**: Tungsten Phase 2 places intermediate data into CPU registers.
  • Loop unrolling and SIMD: Optimize Apache Spark’s execution engine to take advantage of modern compilers and CPUs’ ability to efficiently compile and execute simple for loops (as opposed to complex function call graphs).

Memory Management and Binary Processing

The overhead of JVM objects and GC becomes non-negligible:

  • Java objects have a huge inherent memory overhead
    A typical “abcd” string takes about 24 bytes, use the java object layout tool to check:

  • GC needs estimate the life cycle of objects, but currently it is based on heuristics and estimation. Spark understands how data flows through various stages of computation and the scope of jobs and tasks. Spark should be able to manage memory more efficiently than the JVM

Spark introduces an explicit memory manager to convert most Spark operations to operate directly against binary data rather than Java objects.

The memory manager builds on sum.misc.Unsafe which exposes C-style memory access. Furthermore, Unsafe methods are intrinsic which means that each method call is compiled by JIT into a single machine instruction.

Cache-aware Computation

Spark can leverage the memory resources on a cluster efficiently, processing data at a rate much higher than disk-based solutions. In addition, Spark can also process data orders magnitude larger than the available memory, transparently spill to disk and perform external operations such as sorting and hashing.

Similarly, cache-aware computation improves the speed of data processing through more effective use of L1/ L2/L3 CPU caches, as they are orders of magnitude faster than main memory.

Since most distributed data processing can be boiled down to a small list of operations such as aggregations, sorting and join. By improving the efficiency of these operations, we can improve the efficiency of Spark applications as a whole.

Code generation

At runtime, Spark dynamically generates bytecode for evaluating expressions, rather than stepping through a slower interpreter for each row. Code generation reduces the boxing of primitive data types and more importantly avoids expensive polymorphic function dispatches.

Code generation could speed up many TPC-DS queries as well as built-in expressions. In addition, Spark plans to increase the level of code generation from record-at-a-time expression evaluation to vectorized expression evaluation, leveraging JIT’s capabilities to exploit better instruction pipelining in modern CPUs so we can process multiple records at once.

Also code generation can be used to speed up the conversion of data from in-memory binary format to wire-protocol for shuffle.

Hand-written code is written specifically to run that query and nothing else, and as a result it can take advantage of all the information that is known, leading to optimized code that eliminates virtual function dispatches, keeps intermediate data in CPU registers, and can be optimized by the underlying hardware.

Whole-stage code generation

Whole-stage code generation is about to automatically generate the handwritten code at runtime so Spark engine can achieve the performance of handwritten code, yet provide the functionality of a general purpose engine.

Rather than relying on operators for processing data at runtime, these operators together generate code at runtime and collapse each fragment of the query, where possible, into a single function and execute that generated code instead. For complicated queries, Spark will generate multiple stages code.

Vectorization

When whole-stage code generation is infeasible to generate code to fuse the entire query into a single function, Spark introduces the vectorization.

Instead of processing data one row at a time, vectorization batches multiples rows together in a columnar format and each operator uses simple loops to iterate over data within a batch.

Each next() call would thus return a batch of tuples, amortizing the cost of virtual function dispatches. These simple loops would also enable compilers and CPUs to execute more efficiently with the benefits mentioned earlier.

As a result, Spark uses vectorization only when it is not possible to do whole-stage code generation because vectorization still requires putting intermediate data in-memory rather than keep them in CPU registers.
