概览

集群包含：

  • Master，用于管理Worker节点

  • Worker，存在一个或多个ExecutorBackend进程，每个ExecutorBackend进程包含一个Executor对象，Executor持有一个线程池，每个线程可以执行一个Task

  • Driver

    The process running the main() function of the application and creating the SparkContext.

Spark在执行Job时，首先为建立逻辑执行图（数据依赖图），然后划分逻辑执行图并生成DAG物理执行图，最后根据DAG物理执行图生成具体的Task去执行。

逻辑执行图

生成逻辑执行图（General logical plan）需要清楚数据之间的依赖关系。RDD之间有的依赖关系分为五种：

  • OneToOneDependency
  • N:1 NarrowDependency
  • N:N NarrowDependency
  • RangeDependency（只在UnionRDD中使用）
  • ShuffleDependency

前四种都属于NarrowDependency，注意Spark论文中的定义为：

Each partition of the parent RDD is used by at most one partition of the child RDD

这里的N:N NarrowDependency是该定义并不包含的，但是Spark中可以出现。

关键是来自Parent RDD的数据是否为完整的Partition，如果是，那么就是NarrowDenpendency，否则为ShuffleDependency。

物理执行图

物理执行图会根据Dependency将逻辑执行图划分成多个Stages和Tasks，从而真正执行。

基本思路：从后往前推算，当遇到ShuffleDependency时，则为该Stage的边界，遇到的NarrowDependency运算都加入到该Stage中。每个Stage里的Task数量由该Stage中最后一个RDD的partition数量决定。

注意Stage的编号是从后往前依次增加，如果某个Stage产生了结果，那么其最后一个Task为ResultTask，否则为ShuffleMapTask。

执行

物理执行图的生成是从后往前，而执行则是从前往后。

每个RDD包含getDependency()用于获取该RDD的数据依赖，之后使用compute()方法负责接收到来自parent RDDs或data block流入的records并进行计算，最后输出records。

用户的Driver中的每一个action()都会生成一个对应的Job，每个job将由DAGScheduler根据物理执行图划分成多个Stage，其中最后一个Stage产生最后的结果。在提交job的过程，会根据Stage的依赖关系进行提交。

Shuffle过程

Spark Shuffle与MapReduce Shuffle对比：

  • High-level上看，都是将Mapper的输出按照partition传输到不同的Reducer上。
  • MapReduce Shuffle是sort-based，进入combine()和reduce()的records需要进行排序。但Spark是hash-based，不会对数据提前进行排序，除非显式调用类似sortByKey()等操作。
  • 实现上，MapReduce Shuffle将整个流程划分出明显的几个阶段：map, spill, merge, shuffle, sort, reduce。而Spark中没有明确的阶段划分，都是蕴含在各种transformation()中。

整个Shuffle可以分为两个阶段：

  • Shuffle write，Mapper划分数据，持久化数据的过程
  • Shuffle read，Reducer读入数据，聚集数据的过程

整个过程如下：

Shuffle Write

Shuffle write的逻辑在ShuffleMapTasks中实现，其输出的final RDD（通常为MapPartitionsRDD）会将输出的每一个record进行partition并持久化。Shuffle write持久化的目的：一是减小内存占用，二是实现fault-tolerance。

对于计算出的record，首先会写入到内存中的bucket。bucket的数量与Reducer的数量相同，其大小由spark.shuffle.file.buffer.kb确定，默认为32KB。record写入的bucket则由partitioner.partition(record.getKey())决定。

每个bucket中的数据会不断地写入到磁盘的ShuffleFile文件中，形成一个FileSegment（ShuffleBlock）。为了减少ShuffleFile的数量，同一个核中的连续的ShuffleMapTask可以以追加地方式写入到一个ShuffleFile中，因此文件个数约为cores * R。在Reduce阶段，Reducer会去主动抓取对应ShuffleFile中的records。

Shuffle Read

Shuffle Read会从各个ShuffleMapTask输出的ShuffleFile文件抓取Shuffle Write写入的records。只有当上个Stage的所有ShuffleMapTask都完成时，才会开始抓取。抓取的FileSegement会使用内存进行缓冲，缓冲区softBuffer的大小由spark.reducer.maxMbInFlight设定，默认为48MB。

由于Spark的Reduce阶段并不要求进行排序，所有会一边抓取一边处理。通过一个类似HashMap的数据结构来保存具有相同key的记录，如果给定的key已经具有值，那么调用combine函数进行合并。在ShuffleMapTask产生final RDD时，会通过MapOutputTrackerMaster对写到FileSegment的数据进行注册：MapOutputTrackerMaster.registerShuffle(shuffleId, rdd.partitions.size)。

抓取的FileSegment会首先存放在softBuffer中，经过处理的数据可以放在内存和磁盘上。spark.shuffle.spill用于控制是否存放到磁盘中，在内存中使用的一种AppendOnlyMap，而内存和磁盘则使用ExternalAppendOnlyMap用于保存。

架构

Spark的架构如下：

Job提交

Driver提交Job并运行的步骤为分：

  1. dagScheduler根据用户代码生成Job，代码中的每个action都会产生一个Job。针对每个Job，在dagScheduler.runJob()中分解为多个Stage，每个Stage在dagScheduler.submitStage()又生成多个包含ShuffleMapTasks或ResultTasks的tasks，并将tasks打包成TaskSet。
  2. 将生产的taskSet添加到taskScheduler中（通常为FIFOScheduler）
  3. 通过sparkDeploySchedulerBackend发送task到dagScheduler指定的worker node上CoarseGrainedExecutorBackend的Executor中执行。

Task运行

执行的主要步骤：

  1. Executor先执行收到的task，获得directResult。
  2. 将task结果发送回Driver端：如果directResult的大小小于spark.akka.framesize那么就直接发送，否则directResult将通过blockManager进行存储，将存储位置信息（indirectResult）发送回Driver。
  3. Driver收到task的执行结果后，会根据不同的task类型进行处理。如果task类型为ResultTask，那么将会利用ResultHandler对收到的结果进行汇总和计算， 得到最终的结果；如果task类型为ShuffleMapTask，那么结果为MapStatus，Driver会将这些MapStatus注册到mapOutputTrackerMaster中，以方便reducer shuffle时查询。另外，如果收到的task为Stage中的最后一个task，那么会通知dagScheduler提交下一个Stage，类似的，如果是最后一个Stage，就会通知dagScheduler job运行完成。

需要注意的细节：

  1. 每个Executor中的BlockManager可以将结果存储在内存加磁盘中。BlockManager内部包含一个memoryStore来存储数据在内存中，其占用的内存不超过Runtime.getRuntime.maxMemory * spark.storage.memoryFraction(default 0.6)；如果memoryStore无法存储新的数据，则会将部分数据通过diskStore持久化到磁盘中。
  2. ShuffleMapTask的执行结果为MapStatus，一个MapStatus包括对应task的BlockManager的BlockManagerId（executorId + host, port, nettyPort），以及产生的每个FileSegment的大小。BlockManager中的shuffleBlockManager会为ShuffleMapTask提供shuffleWriters。而ResultTask则是在RDD的partition上的执行结果。

Shuffle Read

Reducer需要抓取数据时，会调用blockStoreShuffleFetcher去获取Mapper产生的FileSegments的位置。blockStoreShuffleFetcher则会调用MapOutputTrackerWorker会利用mapOutputTrackerMasterActorRef与Driver中的mapOutputTrackerMasterActor进行通信，获取MapStatus信息，提取FileSegments的信息为`blocksByAddress: Array[(BlockManagerId, Array[(BlockId, Size(FileSegment))])]。

之后，blockStoreShuffleFetcher会利用BasicBlockFetcherIterator去获取实际的FileSegment的数据。BasicBlockFetcherIterator会产生多个fetchRequest，每个fetchRequest包含去某个节点获取若干个FileSegments的任务。每个Reducer会启动5个并行的线程去执行fetchRequest，能够缓冲的在内存中的总数据大小由spark.reducer.maxMbInFlight(default: 48MB)控制，此外，如果FileSegment的数据在本地节点，则会使用本地读取的方式。抓取的FileSegment以iterator的形式提供给rdd.compute()。

Cache和Checkpoint

Spark提供了内存Cache机制用于缓存被多次访问的数据，从而提高运行速度。

由于Spark的task中computing chain相比Hadoop MapReduce会更加庞大，为了避免task出错导致整个computing chain进行重新计算，Spark会对计算较大的RDD进行checkpoint。

Cache机制

在Driver中，用户可以通过RDD.cache()来显式的告诉Spark哪些RDD需要Cache，在计算该RDD partition时，会直接将结果保存在BlockManager的memoryStore中。

在调用RDD.cache()后，该RDD就会成为persistRDD，其StorageLevel为MEMORY_LEVEL。被缓冲的RDD在下次使用时直接从BlockManager的memoryStore中读取。

Checkpoint

Spark会对消耗运算资源较大，computing chain过长或依赖较多的RDD进行checkpoint，以避免重新计算。checkpoint在Job执行完以后，单独启动一个专门的Job去执行checkpoint。

需要checkpoint的RDD有下面的几个阶段：

  • Initialized：Driver调用rdd.checkpoint()显式指定RDD需要checkpoint，并设置checkpoint的存储路径。之后，该RDD由RDDCheckpointData管理

  • Marked for checkpointing：初始化完成以后，RDDCheckpointData将会标记该RDD为MarkedForCheckpoint

  • Checkpointing in progress：每个Job运行结束后，会调用finalRDD.doCheckpoint()。doCheckpoint()会沿着computing chain回溯，遇到被标记的RDD就会将其状态转换为CheckpointingInProgress，并会将配置文件广播到其他worker节点的blockManager。
    完成以后，会启动一个Job来完成checkpoint：

  • Checkpointed：当RDD完成checkpoint后，会将RDD的dependency清除掉（避免重新计算），并设置RDD的状态为Checkpointed。

区别和比较

  • Cache的RDD存储在内存中，并且保存其的lineage（computing chain）
  • Checkpoint的RDD存储在HDFS上， 会清楚其的lineage

Broadcast

Broadcast是将只读变量在节点上进行缓存，从而使得该节点上的task可以快速的访问。

实现

在Driver端：

  1. 针为需要broadcast的数据建立一个本地文件夹，并启动一个可以访问该文件夹的HttpServer(HttpBroadcast，TorrentBroadcast不同)
  2. 在调用bdata = sc.broadcast(data)时，会将数据写入到本地文件夹中，并存储到blockManager中，其中bdata类型为Broadcast

在Executor端：

  1. 在反序列化时发送到Executor执行的task时，会调用bdata的readObject()方法
  2. readObject()会根据首先检查Executor所在节点的blockManager是否已经存储了该数据，否则则会通过HttpBroadcast/TorrentBroadcast去从其他节点抓取数据

HttpBroadcast

Driver和Executor启动时，都会创建一个broadcastManager对象用于管理broadcast的变量。HttpBroadcast则是创建一个HttpServer，使得其他节点能够抓取数据。缺点是Driver节点的网络负载较高，存在单点网络瓶颈问题，可能出现网络拥堵。

TorrentBroadcast

TorrentBroadcast基于Torrent技术，将data分块为多个data blocks。当其他节点抓取了一个data block以后，自身也成为了该data block的server。

在Driver端，根据由spark.broadcast.blockSize设定的BLOCK_SIZE对数据进行切分，形成多个byteArray。之后，会将分块信息和data block数据存放到blockManager中。

在Executor端，在反序列化时，TorrentBroadcast.readObject()首先在本地的blockManager查询，如果不存在，则会启动BT过程。

TorrentBroadcast存在的问题时慢启动和占用更多的内存空间。
