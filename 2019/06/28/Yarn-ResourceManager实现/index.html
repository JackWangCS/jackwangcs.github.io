YARN采用了Master/Slave结构，其中的Master为ResourceManager，Slave为NodeManager。ResourceManager是YARN集群中最重要的组件，RM的功能包括：ApplicationMaster的管理、NodeManager的管理、Application的管理、状态机管理等。

概述

RM的RPC协议：

  • ResourceTrackerProtocol，NM与RM之间采用了“pull”模型，NM通过周期性地RPC协议注册、汇报Container的状态，并通过返回结果获取RM的命令。
  • ApplicationMasterProtocol，应用程序的AM通过RPC协议向RM注册、申请资源和释放资源，依然采用了“pull”模型。
  • ApplicationClientProtocol，客户端通过RPC协议向RM提交应用程序、查询应用程序状态和控制应用程序。

此外，RM还负责资源管理与调度。

内部架构

RM内部主要分为：

  • 用户交互模块：
    • ClientRMService，处理来自客户端的RPC请求
    • AdminService，为管理员提供的服务接口用于管理集群
    • WebApp，用于通过Web展示集群资源使用情况和应用程序运行状态信息等
  • NM管理模块
    • NMLivelinessMonitor，监控NM是否存活，将超过一定时间（默认10分钟）的未汇报心跳信息的NM进行移除
    • NodesListManager，维护正常节点和异常节点列表，动态加载和管理exclude和include节点列表
    • ResourceTrackerService，处理NM的注册和心跳等请求，NM启动时会向RM进行注册，上报NM包含的节点ID、可用的资源上限等信息；NM的周期性的心跳信息则包含各个Container的运行状态，运行的应用程序列表、节点的健康状况等信息。RM在响应时会返回NM待释放的Container列表、应用程序列表等。
  • AM管理模块
    • AMLivenessMonitor
    • ApplicationMasterLauncher，与某个NM通信，启动应用程序的AM
    • ApplicationMasterService，处理来自AM的注册和心跳等请求
  • Application管理模块
    • ApplicationACLsManager，管理应用程序的访问权限，包括查看权限和修改权限
    • RMAppManager，管理应用程序的启动和关闭
    • ContainerAllocationExpirer，用于管理分配给AM的Container，回收已经超时的Container
  • 状态机管理模块
    • RMApp，维护了一个应用程序的整个运行周期
    • RMAppAttempt，维护了一个应用程序的一次运行Attempt的整个生命周期
    • RMContainer，维护了一个Container的运行周期
    • RMNode，维护了一个NM的生命周期
  • 安全管理模块，由ClientToAMSecretManager，ContainerTokenSecretManager，ApplicationTokenSecretManager等完成
  • 资源分配模块，资源调度器ResourceScheduler按照一定的约束条件将集群中的资源分配给各个应用程序。YARN提供了一个批处理资源调度器FIFO和两个用户调度FairScheduler和CapacityScheduler实现。

ApplicationMasterLauncher

  • CLEANUP，与对应的NM通信，要求杀死AM

launcherHandlingThread从masterEvents阻塞队列中不断的获取需要执行的LauncherRunnable任务，然后通过launcherPool执行。

AMLauncher

AMLauncher在run()方法中，根据不同的事件类型运行：

launch()方法将要启动的AM的ApplicationSubmissionContext封装成StartContainerRequest请求，向对应的NM发送请求，启动运行AM的Container。

对于cleanup()，则是向对应的NM发送stopContainer()请求。

AMLivelinessMonitor

周期性地遍历管理的AM，检查AM是否在一段时间（yarn.am.liveness-monitor.expiry-interval-ms)内未汇报心跳信息。如果超过这段时间，则会认为AM管理的Container运行失败，RM不会重新执行这些Container，而是通过心跳机制告诉对应的AM，由AM决定是否重新执行。如果AM运行失败，RM会重新申请资源，以便分配到一个新的节点上重新执行（重试次数由yarn.resourcemanager.am.max-attempts决定）。AMLivelinessMonitor继承自AbstractLivelinessMonitor，对于超时的ApplicationAttemptId进行利用传入的EventHandler进行处理：

在AbstractLivelinessMonitor中，利用一个checkerThread周期性的检查running状态中保存的AM是否超时，如果超时就移除。

ApplicationMasterService

实现了ApplicationMasterProtocol协议，负责处理来自AM的注册、心跳和清理请求。启动时，通过ApplicationMasterProtocol#registerApplicationMaster()进行注册。在AM运行过程中，通过ApplicationMasterProtocol#allocate()函数周期性地与RM通信，用于：

  • 请求资源
  • 获取新分配的资源
  • 汇报周期性的心跳

AM运行结束时，通过ApplicationMasterProtocol#finishApplicationMaster()通知RM，从而回收资源（AM所在的Container）和清理各种数据结果。

NodeManager管理

由NMLivenessMonitor、NodesListManager和ResourceTrackerService组成。NMLivenessMonitor继承自AbstractLivenessMonitor，与AMLivenessMonitor实现类似。

NodesListManager

NodesListManager管理exclude和include节点列表，此外两个列表可以通过adminService的bin/yarn rmadmin -refreshNodes动态刷新和加载两个文件。NodesListManager还实现了EventHandler接口，负责NodesListManagerEvent事件的处理。

ResourceTrackerService

ResourceTrackerService实现了ResourceTracker接口，该接口使用ResourceTracker.registerNodeManager()和ResourceTracker.nodeHeartbeat()分别处理NM的注册和心跳：

注册

NM的注册分为两种，第一种是一个全新的NM进行注册，另外一种是NM的重连导致的注册。在注册前，会根据nodeManagerVersion检查版本信息，之后通过nodesListManager判断是否是有效的节点，然后利用capacity检查NM是否是满足最小内存和vcore数量的NM。之后：

心跳

主要步骤：

  1. 检查是否是注册的节点（registered），unregistered：resync。

  2. 检查是否是有效的节点（valid）-> unvalid：关闭并产生RMNodesEventType.DECOMMISSION事件。

  3. 检查是否是fresh的心跳，而不是重复的心跳。重复心跳，那么直接忽略，并发送上一次的response；如果远程的NM的心跳信息落后，则会产生RMNodeEventType.REBOOTING，并发送resync。

  4. 给RMNode发送NM的healthStatus

Application管理

Application管理涉及应用程序的权限管理、启动与关闭和生命周期等。

ApplicationACLsManager

负责管理应用程序的访问权限：

  • 查看权限，用于查看应用程序基本信息
  • 修改权限，用于修改应用程序优先级、杀死应用程序等

默认情况下，任意一个普通用户可以查看所有其他用户的应用程序。用户可以通过ContainerLauncherContext为自己的应用程序设置具有访问权限的用户列表。访问权限通过applicationACLS的ConcurrentMap保存：

ApplicationACLsManager.checkAccess()首先通过从applicationACLS中获取对应类型的ACL，然后结合adminAclsManager、applicationOwner和ACL来决定用户是否拥有查询的权限：

AdminACLsManager类保存了属于admin的ACL：

RMAppManager

负责应用程序的启动和关闭。ClientRMService收到来自客户端的提交应用程序后，调用RMAppManager#submitApplication()创建一个RMApp对象。RMApp将维护这个应用程序的整个生命周期，当RMApp运行结束后，将向RMAppManager发送一个APP_COMPLETED事件。RMAppManager收到该事件后，通过RMAppManager#finishApplication进行RMApp的清理和收尾工作，包括将该应用程序放入已经完成应用程序列表中以及从RMStateStore中移除。

已完成应用程序列表的大小默认为10000（yarn.resourcemanager.max-completed-applications)，当超过该大小以后，将会在内存中移除过期的历史应用程序，使得用户只能从History Server中获取过期的应用程序信息。History Server从磁盘中的应用程序运行日志和基本信息文件中获取应用程序的信息。

RMStateStore记录了运行中的应用程序的运行日志，当集群因故障重启后，ResourceManager可以通过这些日志恢复应用程序运行状态，从而避免全部重新运行所有的应用程序。当应用程序运行结束后，就不需要RMStateStore来保存状态了。

RMApp状态机

实现类为RMAppImpl，维护了一个Application的状态机，记录了Application的存在的各个状态RMAppState以及导致状态间转换的事件RMAppEvent。此外，还保存了Application的基本信息和迄今为止所有的Application Attempt信息。

状态转移

RMApp状态机的状态转移图如下：

基本状态：

  • NEW
  • NEW_SAVING，RM收到用户提交的应用程序后，创建RMAppImpl对象，并通过日志记录该应用程序的基本信息，以便故障重启后可以自动恢复运行。
  • SUBMITTED，提交的应用程序通过合法验证以及完成日志记录后，RM会根据RMAppImpl对象创建一个RMAppAttemptImpl对象，进行第一次运行尝试。
  • ACCEPTED，资源调度器同意接受该应用程序，并对应用程序的资源请求进行合法性检查。
  • RUNNING，应用程序的ApplicationMaster已经在某个节点上开始运行。
  • KILLED
  • FAILED，应用程序经历过yarn.resourcemanager.am.max-attempts次数的重试后，依然运行失败。
  • FINISHING，ApplicationMaster通过ApplicationMasterProtocol#finishApplicationMaster通知RM即将运行结束。
  • FINISHED，通过NM的心跳汇报ApplicationMaster所在的Container运行结束。

RMAppAttempt状态机

实现是RMAppAttemptImpl，除了维护Application Attempt的生命周期外，还包括当前使用的Container信息，AM的tracking URL等。本质上是维护的ApplicationMaster的生命周期。

状态转移

状态转移图如下：

状态：

  • NEW
  • SUBMITTED
  • SCHEDULED，RM将RMAppAttempt资源请求提交给ResourceScheduler中，ResourceScheduler开始为AppAttempt的AM分配资源。
  • ALLOCATED_SAVING，收到来自ResourceScheduler分配的一个Container，RMAppAttempt会将Container信息写入到磁盘上以便故障恢复，当信息保存完成之前，RMAppAttemptImpl将处于该状态。
  • ALLOCATED，分配给AM的Container信息保存到文件中完成。
  • LAUNCHED，RM中的ApplicationMasterLauncher与对应的NM通信，启动AM，转变为LAUNCHED。
  • RUNNING，AM在NM上成功启动，通过ApplicationMasterProtocol#registerApplicationMaster向RM注册，此时RMAppAttemptImpl状态转变为RUNNING。
  • FAILED
  • FINISHED
  • LAUNCHED_UNMANAGED_SAVING
  • RECOVERED

RMContainer状态机

YARN使用Container来对资源进行描述，RMContainer维护了一个Container的生命周期。

状态转移

状态包括：

  • NEW
  • RESERVED，Container包含的资源尚不能满足应用程序需求时，YARN让该节点为其预留资源。
  • ALLOCATED，ResourceScheduler将一个Container分配给一个应用程序，此时，Container处于可使用状态，但是AM还未获取该Container。
  • ACQUIRED，AM通过ApplicationMasterProtocol#allocate获得RM分配给自己的Container列表，这些Container状态被设置为ACQUIRED。
  • RUNNING，AM与对应的NM通信，启动Container。之后，NM通过心跳机制将Container的状态汇报给RM，RM将该Container的状态设置为RUNNING。
  • RELEASED，AM通过ApplicationMasterProtocol#allocate向RM发送请求时，会释放部分Container。RM收到释放请求后，会将这些Container的状态设置为RELEASED，并向RMNodeImpl发送RMNodeEventType.CLEAN_CONTAINER事件来清理该Container。
  • COMPLETED，NM通过ResourceTracker#nodeHeartbeat告诉RM已经运行完成的Container列表，RM收到信息后会将这些Container的状态设置为COMPLETED。
  • EXPIRED，AM在获得Container分配后，在一定时间内并未使用该Container而超时。
  • KILLED

RMNode状态机

RMNode维护了一个节点的状态机，实习为RMNodeImpl。状态转移如下：

状态：

  • NEW
  • RUNNING，NM启动后，通过ResourceTracker#registerNodeManager向RM注册，进入RUNNING状态。
  • UNHEALTHY，每个NM上可以配置一个健康状况监控脚本，用于周期性检查NM的健康状态。如果RM发现NM不健康，则会转变到UNHEALTHY状态。处于该状态的节点将不会被分配新的任务，直到变为健康状态为止。
  • DECOMMISSIONED，节点被加入到exclude中，NodeManager将会被设置为DECOMMISSIONED状态，使得NM无法与RM通信。
  • LOST，如果NM在一定时间间隔没有汇报心跳消息，则认为该节点处于LOST状态，它上面正在运行的所有Container将会被设置为FAILED状态。
  • REBOOTED，如果RM发现NM汇报的心跳ID与自己保存的不一致，则会置为REBOOTED状态，从而重新启动完成同步。
