This chapter focus further on the data processing patterns.

Roadmap

Except event time & processing time and windowing, we are now going to look closely at:

  • Triggers

    A trigger is a mechanism for declaring when the output for a window should be materialized relative to some external signal. Triggers also make it possible to observe the output for a window multiple times as it evolves.

  • Watermarks

    A watermark is a notion of input completeness with respect to event times. A watermark with value of time X makes the statement: “all input data with event times less then X have been observed.”

  • Accumulation

    An accumulation mode specifies the relationship between multiple results that are observed for the same window.

Four questions that are critical to every unbounded data processing problem:

  1. What results are calculated? This question is answered by the types of transformations within the pipeline.
  2. Where in event time are results calculated? This question is answered by the use of event-time windowing within the pipeline.
  3. When in processing time are results materialized? This question is answered by the use of triggers and (optionally) watermarks.
  4. How do refinements of results relate? This question is answered by the type of accumulation used: discarding, accumulating, or accumulating and retracting.

Batch Foundations: What and Where

What: Transformations

The transformations applied in classic batch processing can answer “What result are calculated?” question. The book uses Apache Beam to define the pipeline, there are two basic primitives in Beam:

  • PCollections, represent datasets across which parallel transformations can be preformed.

  • PTransforms are applied to PCollections to create new PCollections. PTransforms has three kinds of transformations: element-wise, group/aggregate, composite.

Where: Windowing

Windowing is the process of slicing up a data source along temporal boundaries. For the batch engine, there is only one global window.

Going Streaming: When and How

When: Triggers

Triggers provide the answer “When in processing time are results materialized?”. Each specific output for a window is referred to as a pane of the window.

There are generally two types of triggers:

  • Repeated update triggers which periodically generate updated panes for a window as its content evolves. The choice of period for a repeated update trigger is an exercise in balancing latency and cost.
  • Completeness triggers which materialize a pane for a window only after the input for that window is believed to be complete to some threshold.

Per-record triggering is quite chatty, we can use the processing-time delays to equalize effect across high-volume keys or windows: the resulting stream ends up being more uniform cardinality-wise.

There are two approaches to processing-time delays in triggers:

  • aligned delays where the delay slices up processing time into fixed regions that align across keys and windows. For example: the micro-batch streaming system like Spark Streaming
    • Predictability
    • All updates happen at once which results in bursty workloads
  • unaligned delays where the delay is relative to the data observed within a given window

When: Watermarks

Watermarks are a supporting aspect of the answer to the question: “When in processing time are results materialized?”

Watermarks are temporal notions of input completeness in the event-time domain. Watermark can be seen as a function $F(P) \to E$: which takes a point in processing time and returns a point in a event-time. It’s an assertion that no more data with event times less than $E$ will ever be seen again.

  • Perfect watermarks, all data are early or on time
  • Heuristic watermarks which use whatever information is available about inputs to provide an estimate of progress that is as accurate as possible

Watermarks form the foundation for completeness triggers. In most cases, windows are materialized as the watermark passes the end of window.

Watermark gives us a way to reason about the completeness of our input which is especially important for lack-data in input or missing data cases

Shortcomings of watermark:

  • Too slow, when a watermark is correctly delayed due to known unprocessed data that translates directly into delays in output. For example, the 2nd and 3rd windows are delayed even though all events have arrived before the 9 event in the first window in perfect watermark figure.

    Depending upon completeness for producing output is often not ideal from a latency perspective.

  • Too fast, when a heuristic watermark is incorrectly advanced earlier than it should be. Relying on heuristic watermarks alone for determining when to materialize output is insufficient if you care about correctness.

When: Early/On-Time/Late Triggers FTW

The combination of repeated update triggers and completeness/watermark triggers is for the win which is called early/on-time/late trigger.

It partitions the panes that are materialized by the compound trigger into three categories:

  • Zero or more early panes which are the result of a repeated update trigger that periodically fires up until the watermark passes the end of the window. This compensates for the watermark too slow shortcoming.
  • A single on-time pane, which is the result of the completeness/watermark trigger firing after the watermark passes the end of the window. This means that it is now safe to reason about missing data.
  • Zero or more late panes which are the result of another repeated update trigger that periodically fires any time late data arrive after the watermark has passed the end of the window. This compensates for the watermark too fast shortcoming.

The biggest remaining difference between the perfect and heuristic early/on-time/late versions is the window lifetime bounds at this point. In the heuristic watermark case, we need to hold on to the state for a window for some amount of time to account for the late data.

When: Allowed Lateness(GC)

It’s not practical to keep state for a given window indefinitely so any real-world out-of-order processing system needs to provide some ways to bound the lifetime of the windows it’s processing.

One way is to define a horizon on the allowed lateness within the window.

How: Accumulation

Accumulation is the answer to the question: “How do refinements of results relate”. Three types of accumulation:

  • Discarding: Useful when the downstream consumer itself is performing some sort of aggregation on materialized panes

  • Accumulating: Useful when you can simple overwrite previous values with new values, the new values already incorporates all of the data seen thus far

  • Accumulating and retracting. Each pane includes both a new accumulating mode value as well as retraction of the previous pane’s value.

    Retractions are essentially an explicit way of saying “I previously told you that the result was X, but I was wrong. Get ride of the X and replace with Y”. Retractions are particularly helpful when:

    • When consumers downstream are regrouping data by a different dimension, new data may end up keyed in a different group.
    • When dynamic windows are in use, the new value might replace more than one previous window due to window merging

Summary

  • What results are calculated? transformations
  • Where in event time are results calculated? windowing
  • When in processing time are results materialized? triggers + watermarks
  • How do refinements of results relate? accumulation
