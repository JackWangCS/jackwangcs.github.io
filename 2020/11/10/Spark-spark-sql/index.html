Spark SQL模块用于结构化的数据处理，Spark SQL相对于RDD对数据和执行的计算拥有更多的结构信息，从而可以执行额外的优化。用户通过统一的SQL和DataFrame接口和Spark SQL进行交互。

DataSet和DataFrame

定义

Dataset是在Spark 1.6引入，在RDD的基础上加强了类型约束，从而能够被Spark SQL的执行引擎使用。正式的定义如下：

A Dataset is a strongly typed collection of domain-specific objects that can be transformed in parallel using functional or relational operations.

而每一个Dataset都具有一个untyped view，称为DataFrame，类型为Dataset[Row]。Row则表示了关系操作符的一行输出（one row of output from a relational operator），可以通过Row.apply()来创建，可以通过序号来访问列值或者通过基础类型接口来获得列值。

同样地，Dataset的操作可以分为transformation和action，并且Dataset也是懒计算的（lazy）。在内部，Dataset表示一个描述了生产数据需要的计算的逻辑计划（A Datasetrepresents a logical plan that describes the computation required to produce the data）。当一个action被调用，Spark的Query Optimizer将会优化该逻辑计划，并生成能够用于执行的高效的、并行的、分布式的物理执行计划，从而获得实际的数据。用户可以通过explain()函数来查看逻辑计划以及优化后的逻辑计划。

为了支持特定领域（domain-specific）的对象，DataSet需要一个Encoder，用于将类型T映射为Spark内部类型系统。通常，基于Spark的内部类型系统表示的二进制结构相对于原始的Object对象具有更小的内存消耗（footage），可以通过schema()函数来查看。

综合上面的两部分，Dataset类的定义为：

QueryExecution是对LogicalPlan和SparkSession的封装，表示执行关系型查询的工作流。一个Query Planning的过程包含四个阶段：Parsing，Analysis，Optimization和Planning，最终将LogicalPlan转换成一个SparkPlan（物理执行计划）。

Encoder接口的定义如下：

一个JVM类型T，可以用StructType来描述其类型信息。Spark利用StructType将类型T转换成Spark SQL的内部表示，或者反之。StructType由StructType或者基础的StructFeild组合而成，利用了组合设计模式。目前，Encoder的唯一实现是ExpressionEncoder，ExpressionEncoder在构造时需要用于序列化和反序列化的Expression：

SparkSession

SparkSession是利用Spark SQL进行编程的入口点（entry point），除了持有SparkContext引用，并提供了额外的SQLContext用于SparkSQL模块。

通过SparkSession，用户可以创建DataFrame，读取多种数据源（read和readStream），注册UDF，执行SQL语句。所有中间数据类型都是DateFrame。

SharedState负责存储在给定SQLContext下的共享状态，主要包括：

  • 加载hive-site.xml，初始化warehousePath，SparkConf和HadoopConf
  • CacheManager
  • ExternalCatalog
  • GlobalTempViewManager
  • jarClassLoader

SessionState在包含了所有session相关的状态：

  • SharedState
  • SQLConf
  • ExperimentalMethods，用于添加自定义的planning策略和优化器的接口
  • FunctionRegistry，用于Internal Catalog管理用户注册的函数的接口
  • UDFRegistration，用于添加UDF的接口
  • SessionCatalog，创建Internal Catalog的Builder
  • ParserInterface，从SQL语句中解析表达式，表名等
  • Analyzer，Logical Plan分析器，解析未解析的属性和关系
  • Optimizer，Logical Plan优化器
  • SparkPlanner，将优化后的Logical Plan转换为Physical Plan
  • StreamingQueryManager，Streaming查询管理器
  • SessionResourceLoader
  • QueryExecution：根据Logical Plan创建QueryExecution，相当于对逻辑计划进行优化，
