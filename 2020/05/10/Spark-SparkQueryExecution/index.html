Spark Query Execution

Spark3.0的架构如下：

SQL、Spark ML、Spark Streaming等都基于底层统一的SQL Engine。整个流程如下：

  • Parse：解析SQL语句，得到语法树并生成Unresolved Logical Plan
  • Analysis：利用Session Catalog信息，解析Logical Plan的数据库名，表名， 函数等信息
  • Logical plan Optimizations：利用Optimizer优化Logical Plan
  • Physical planning：将Logical plan转换成多个Physical plan，并根据Cost Model选择最近的Physical plan
  • Code Generation：将Physical Plan中的部分生成Java字节码，从而提高执行效率
  • Execution：基于Spark Core执行底层RDD计算

前三部也称为Front-end，后部分为backend。

Parse

Spark 2.0以后引入了ANTLR语法解析工具来解析SQL语句，常见的大数据组件如Hive，Cassandra，Pig和Presto等都是用ANTLR来解析SQL。Spark基于Presto定义了SparkSQL的语法文件SqlBase.g4(sql\catalyst\src\main\antlr4\org\apache\spark\sql\catalyst\parser\SqlBase.g4)。根据SqlBase.g4，利用ANTLR可以生成词法分析器SqlBaseLexer.java和语法分析器SqlBaseParser.java。

词法分析器用于机械关键词和各种标志符，语法分析器用于分析语法并构建语法树。

Spark会利用AstBuilder将语法树转换成Logical Plan，此时的Logical Plan因为没有Catalog信息，因此是Unresolved Logical Plan。

Analysis

Analysis阶段会使用事先定义好的 Rule 以及 SessionCatalog 等信息对 Unresolved LogicalPlan 进行 转换，得到Analyzed Logical Plan。SessionCatalog 主要用于各种函数资源信息和元数据信息（数据库、数据表、数据视图、数据分区与函数等）的统一管理。

Analyzer的定义如下：

Analyzer继承了RuleExecutor框架，只需要定义和实现各种Rule Batches即可。

RuleExecutor

RuleExecutor用于将多个Rule进行组合，从而将一个Plan转换成具有同样类型的另外一种Plan。Batch是Rule的集合，具有对应的执行Strategy：

每个Strategy描述了Rule执行的终止条件，即通过maxIterations来定义最大的执行次数。Spark提供了Once, FixedPoint两种Strategy。

RuleExecutor执行时，会一次按照Batch的顺序进行执行，根据各个Batch的不同Strategy来决定是否完成当前Batch的执行，从而执行下一个Batch。

Logical Plan Optimizations

目前，Spark的优化器Optimizer主要是基于规则的优化（Rule-Based Optimizer，RBO），绝大部分的规则都是启发式或者基于直观经验得出的。Optimizer的定义如下：

也是继承了RuleExecutor，定义用于优化的Rule Batches。默认提供的defaultBatches主要包括：

  • Operator Push Down
    • PushProjectionThroughUnion
    • ReorderJoin
    • PushDownPredicates
    • LimitPushDown
    • ColumnPruning
    • ….
  • Operator Combine
    • CollapseProject, CollapseWindow, CollapseRepartition
    • CombineFilters, CombineLimits, CombineUnions
  • Constant folding and strength reduction
  • Eliminate Distinct
  • Join Reorder: CostBasedJoinReorder
  • Rewrite Subquery
  • …

Physical planning

将Logical plan转换成多个Physical plan，并根据Cost Model选择最近的Physical plan。Spark Planner包括两个阶段：

  1. 将Logical Plan根据Strategies转换为Physical Plan
  2. 利用Rule Executor转换Physical Plan使得其能够执行
