Flink Deployment on Cluster

Flink Cluster的架构如下：

Deployment Modes

Flink根据不同Resource Provider的资源分配行为，高可用和支持的Job提交方式的不同，提供了三种实现。支持的部署模式如下：

  • Per-Job Mode：单个应用独占一个Flink集群。Job的main()方法在集群创建之前在Client端执行。缺点：对Client的网络和计算资源都有一定的要求，对共享Client的情况不太友好。Client端需要从网络中下载Job的各种依赖，并在本地解析应用的逻辑，获取JobGraph表示；之后，还负责将JobGraph和相关的依赖上传到集群中。
  • Session Mode：一个JobManager实例管理多个Job，这些Job共享TaskManagers。
  • Application Mode：单个应用独占一个Flink集群。Job的main()方法在JobManager中执行。Application Mode可以看做只有一个Job的Session Mode。Job可以多次调用execute()/executeAsync()方法，但是只有单个execute()方法时，才支持HA。

三种模式的不同之处主要有：

  • Flink集群的生命周期
  • 资源隔离级别保证
  • 是否Application的main()方法在Client或者在集群中执行

JobManager

JobManager是Flink分布式执行的Master组件，负责调度任务，协调Checkpoints和实现错误恢复等。JobManager进程主要包含三个组件：

  • ResourceManager：负责Flink集群中资源的分配/释放以及供应
  • Dispatcher：提供了用于提交Flink Applications执行的REST接口，对于每一个提交的Job，会启动一个单独的JobMaster负责执行。此外，Dispatcher也会运行Flink WebUI用于展示Flink Job的执行信息。
  • JobMaster：负责管理单个JobGraph的执行

ClusterEntryPoint中包含了与具体部署模式无关的服务，主要有：

  • Metrics相关：MetricRegistryImpl和ProcessMetricGroup

DispatcherResourceManagerComponent

DispatcherResourceManagerComponent主要包含：

  • DispatcherRunner：用于执行Dispatcher，Dispatcher负责处理Job的提交、持久化提交的Job、创建JobMaster并执行Job以及Job的错误恢复等。
  • ResourceManagerService：用于管理ResourceManager的生命周期。ResourceManager负责对资源的申请、释放和管理。
  • dispatcherLeaderRetrievalService和resourceManagerRetrievalService：负责Dispatcher和ResourceManager的HA
  • webMonitorEndpoint：JobManager的Web UI服务

ResourceManager

单个Flink集群的资源都由ResourceManager管理，ResourceManager可以服务多个JobMaster。ResourceManager抽象类实现了JobManager和TaskManager的注册和注销逻辑，Slot的管理由SlotManager完成。目前有两种子类：

  • StandaloneResourceManager：用于Standalone的Flink集群，在启动后不会请求新资源。
  • ActiveResourceManager：向外部的资源管理框架积极地（actively）请求和释放资源。ResourceManagerDriver负责与具体的外部资源管理框架通信，ActiveResourceManager用于决定何时请求和释放资源，以及请求和释放资源的量。

ResourceManager管理了多个TaskExecutor，TaskExecutor向ResourceManager汇报TaskExecutor上的资源情况，ResourceManager集中管理整个集群的资源。每个JobMaster通过requestSlot() RPC方法向ResourceManager请求新的Slot。

其中，ResourceManagerPartitionTracker用于跟踪在TaskExecutor上的Partition的情况。

SlotManager

SlotManager用于维护所有注册的TaskManager中的Slot，并负责处理新的Slot请求和Slot的分配。当一个新的Slot注册或者已分配的Slot变为空闲时，SlotManager会尝试将该Slot分配给请求Slot的JobManager。如果当前没有足够的Slot，SlotManager将会通过ResourceActions的allocateResource()返回的结果通知JobManager。

TaskManager在启动以后，会通过registerTaskManager()向SlotManager进行注册。完成注册以后，会周期性地通过reportSlotStatus()上报TaskManager的Slot使用情况。

JobMaster中的SlotPool通过registerSlotRequest()向SlotManager请求一个新的Slot，在使用完成以后，会通过freeSlot()通知SlotManager回收该Slot。

SlotManager还会负责检查集群的所有资源是否能够满足JobManager的资源请求。

目前，Flink提供了三种SlotManager实现：

  • SlotManagerImpl：目前的默认实现，是一种Coarse-Grained Slot管理方式。在Flink 1.14中被移除，由DeclarativeSlotManager代替。

  • FinedGrainedSlotManager：由FLIP-56 Dynamic Slot Allocation引入，FinedGrainedSlotManager支持ResourceProfile不同的Slot请求。与目前的Coarse-Grained的Slot管理不同，各个Task可以根据实际情况，细粒度地请求具有不同的资源量的Slot：

  • DeclarativeSlotManager：由FLIP-138 Declarative Resource Management引入，DeclarativeSlotManager允许JobMaster以声明式地方式告知ResourceManager整个Job所需要的最小的，目标的和最大的Slot数量。在默认情况下，JobMaster需要每次通过ResourceManagerGateway#requestSlot(JobMasterId, SlotRequest)请求单个的Slot，如果请求的Slot得不到满足，那么整个Job会失败。在引入Declarative Resource Management引入后，JobManager可以根据实际能够获得Slot数量，调整Job的并行度，从而使得Job在资源不足时，也能具体运行。

    为此，SlotManager接口额外添加了ResourceManagerGateway#declareRequiredResources(JobMasterId jobMasterId, ResourceRequirements resourceRequirements) RPC，用于JobMaster在启动时声明对自己的资源需求。

SlotManagerImpl(Deprecated)

SlotMangerImpl是一种Coarse Grained的Slot管理方式，它假定各个TaskExecutor的总资源是一致的，并且每个单独Slot的资源也相同，因此可以很方便地根据Slot请求计算所需要的TaskManager的个数。

其中：

  • ResourceActions：SlotManager与ResourceManager对资源分配和释放命令的封装。

  • SlotMatchingStrategy：用于从空闲的Slot寻找与请求的Slot最为匹配的Slot：

    默认实现有AnyMatchingSlotMatchingStrategy（默认）和LeastUtilizationSlotMatchingStrategy。LeastUtilizationSlotMatchingStrategy根据各个TaskManager的Slot情况，选择具有最低资源利用率（used slot/ total slot）的TaskManager上的空闲Slot进行分配，由cluster.evenly-spread-out-slots启用。

SlotManager相关的配置：

  • slotmanager.number-of-slots.max：Flink集群允许的Slot的最大值，主要用于Batch Job，用于限制总资源的消耗量。对于Stream Job，不建议配置手动配置该参数，如果配置的值过小，那么Stream Job会因为不能获得足够的Slot而失败。对于Standalone集群，Slot分配的数量不由SlotManager控制。
  • slotmanager.redundant-taskmanager-num：允许的富余的TaskManager的数量。富余的TaskManager可以用来加速Job的恢复。
  • slotmanager.request-timeout：Slot请求超时时间。
  • cluster.evenly-spread-out-slots：控制SlotManager的SlotMatchingStrategy策略。

ResourceManagerPartitionTracker

ResourceManagerPartitionTracker用于在ResourceManager跟踪和释放Partition。在资源释放时，ResourceManager需要知道即将被释放的TaskManager上是否还保留了用于Shuffle的Partition。这就是ResourceManagerPartitionTracker的主要用途。

ClusterPartitionReport记录了集群中TaskExecutor中的所有Partition。

ResourceManager会调用releaseClusterParitions()当用户通过DELETE /dataset/:datasetid REST API进行管理时。

JobMaster

JobMaster负责执行的单个的JobGraph，而JobManagerRunner用于运行JobMaster实例。

JobManagerRunner接口的实现为JobMasterServiceLeadershipRunner，管理了JobMasterServiceProcess的leadership。当获得leadership时，JobManagerRunner会创建新的JobMasterServiceProcess实例；当leadership被撤回时，JobManagerRunner会停止JobMasterServiceProcess。

JobMasterServiceProcess会负责创建JobMasterGateway实例，目前的实现为JobMaster。JobMaster实现了JobMasterGateway和JobMasterService接口：

  • CheckpointCoordinatorGateway：用于TaskManager向CheckpoingCoordinator发送Checkpoint消息
    • acknowledgeCheckpoint()
    • declineCheckpoint()
    • reportCheckpointMetrics()
  • JobMasterOperatorEventGateway：用于TaskManager的Operator向JobManager中的OperatorCoordinator发送OperatorEvent。Operator的RPC框架在FLINK-15099引入，用于实现新的Source接口（FILP-27）
    • sendOperatorEventToCoordinator()
  • KvStateRegistryGateway：用于管理Key-Value state的注册和注销
    • notifyKvStateRegistered()
    • notifyKvStateUnregistered()
  • KvStateLocationOracle：用于TaskExecutor请求Key-Value State的位置信息
    • requestKvStateLocation()

Start & Stop

JobMaster启动时，会调用startJobExecution()会启动JobMaster中各种服务，并开始调度：

JobMaster的停止过程是启动的逆过程，关闭所有启动的服务。

RPC Processing

JobMaster需要处理的RPC请求主要有：

  • Task执行调度：cancel()，updateTaskExecutionState()，requestNextInputSplit()，requestPartitionState()，notifyPartitionDataAvailable()
  • Job信息和状态查询：requestJobDetails()，requestJobStatus()，requestJob()
  • TaskManager管理：registerTaskManager，disconnectTaskManager()，heartbeatFromTaskManager
  • Slot资源管理：offerSlot()，failSlot()，notifyAllocationFailure()，notifyNotEnoughResourcesAvailable()
  • Savepoint：triggerSavepoint()，stopWithSavepoint()
  • ResourceManager：disconnectResourceManager()，heartbeatFromResourceManager()
  • OperatorCoodinator：deliverCoordinationRequestToCoordinator()
  • 其他：udpateGlobalAggregate()

Task执行调度和Job信息状态查询等，都交由SchedulerNG进行处理。SchedulerNG包含了Job执行的所有调度和执行状态信息，此外，Operator Coodinator和Savepoint也由SchedulerNG管理。

TaskManager和Slot管理则主要由SlotPoolService完成，SlotPoolService根据注册的TaskManager，来管理Slot Pool，负责申请和释放在TaskManager上的Slot。JobMaster会管理与ResourceManager的连接，当ResourceManager的状态发生改变时，JobManager将会尝试自动重新连接到ResourceManager。

SlotPoolService

SlotPool处理来自ExecutionGraph的Slot请求。如果没有可用的Slot，SlotPool将会向ResourceManager请求新的Slot。此外，SlotPool将会保留所有ResourceManager提供的Slots，因此，即使当ResourceManager不可用时，依然可以为ExecutionGraph提供空闲的Slot。当所有的Task都已经运行，多余的Slot将会被释放。
