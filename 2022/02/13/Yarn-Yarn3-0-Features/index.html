This is summary about new features in Hadoop 3.0+ versions comparing to Hadoop 2.6.

Yarn Federation

Main components:

  • Router

    YARN applications are submitted to one of the Routers, which in turn applies a routing policy (obtained from the Policy Store), queries the State Store for the sub-cluster URL and redirects the application submission request to the appropriate sub-cluster RM

  • AMRM Proxy

    The AMRMProxy is a key component to allow the application to scale and run across sub-clusters. The AMRMProxy runs on all the NM machines and acts as a proxy to the YARN RM for the AMs by implementing the ApplicationMasterProtocol. Applications will not be allowed to communicate with the sub-cluster RMs directly.

  • Global Policy Generator

    Global Policy Generator overlooks the entire federation and ensures that the system is configured and tuned properly all the time. A key design point is that the cluster availability does not depends on an always-on GPG.

  • State Store

    The Federation State defines the additional state that needs to be maintained to loosely couple multiple individual sub-clusters into a single large federated cluster.

    • Sub-cluster membership
    • Application’s Home Sub-cluster
    • Federation Policy
    • Capacity Allocation acorss sub-clusters(Not implemented yet)

Capacity Scheduler

The CapacityScheduler is designed to run Hadoop applications as a shared, multi-tenant cluster in an operator-friendly manner while maximizing the throughput and the utilization of the cluster.

Features:

  • Hierarchical Queues
  • Capacity Guarantees
  • Security
  • Elasticity
  • Multi-tenancy
  • Operability
    • Runtime configuration: The queue definitions and properties such as capacity, ACLs can be changed, at runtime
    • Drain applications
  • Resource-based Scheduling
  • Queue Mapping Interface based on Default or User Defined Placement Rules
    • Automatically mapping user’s job to queue based on rules
    • Dynamic Auto-Creation and Management of Leaf Queues
  • Priority Scheduling
  • Absolute Resource Configuration: Administrators could specify absolute resources to a queue instead of providing percentage based values.

Queue Absolute Resource Configuration

  1. If absolute resource is configured, sum of absolute resources of child queues could be less than its parent absolute resource capacity. Applications in the queue may consume more resources than the queue’s capacity if there are free resources, providing elasticity.

Application Priority

Application priority works only along with FIFO ordering policy. Default ordering policy is FIFO.

  • Cluster-level priority: yarn.cluster.max-application-priority
  • Leaf queue-level priority: yarn.scheduler.capacity.root.<leaf-queue-path>.default-application-priority

Queue Priority

Simply ordering queues according to relative used-capacities sometime causes a few troubles because scarce resources could be assigned to less-important applications first.
Example: Resource fragmentation for large-container apps:

Queue priorities of all child queues of a parent affect the ordering of queues under the same parent. The queue priority is local to the parent-child relationship. If two queues do not share the same parent, then relative-priority is decided by the direct children of the lowest common ancestor. Resource allocations:

  • Both queues are under their guaranteed capacities: The queue with the higher priority gets resources
  • Both queues are over or meeting their guaranteed capacities: The queue with the higher priority gets resources
  • One of the queues is over or meeting their guaranteed capacities, the other is under: The queue that is under its capacity guarantee gets resources

For resource-preemption, its the resversed way of resource allocations.

Reservation System

The CapacityScheduler supports the following parameters to control the creation, deletion, update, and listing of reservations.

The CapacityScheduler supports the ReservationSystem which allows users to reserve resources ahead of time. The application can request the reserved resources at runtime by specifying the reservationId during submission.

  • yarn.resourcemanager.reservation-system.class: CapacityReservationSystem
  • yarn.resourcemanager.reservation-system.plan.follower: CapacitySchedulerPlanFollower. the class name of the PlanFollower that runs on a timer, and synchronizes the CapacityScheduler with the Plan and viceversa.

Changing Queue Configuration

Changing queue/scheduler properties and adding/removing queues can be done in two ways, via file or via API. Four configuration store implementations:

  • File, allowed to modify via file
  • memory, allowed to modify via API
  • leveldb, allowed to modify via API
  • Zk, allowed to modify via API

For API:

  • yarn.scheduler.configuration.store.class
  • yarn.scheduler.configuration.mutation.acl-policy.class: ACL
  • yarn.scheduler.configuration.store.max-logs: For audit

When enabling scheduler configuration mutations via yarn.scheduler.configuration.store.class, yarn rmadmin -refreshQueues will be disabled, i.e. it will no longer be possible to update configuration via file.

The type of configuration store to use for scheduler configurations.
Default is “file”, which uses file based capacity-scheduler.xml to
retrieve and change scheduler configuration. To enable API based
scheduler configuration, use either “memory” (in memory storage, no
persistence across restarts), “leveldb” (leveldb based storage), or
“zk” (zookeeper based storage).

API based configuration is only useful
when using a scheduler which supports mutable configuration. Currently only capacity scheduler supports this.

Rest API:

  • http://rm-http-address:port/ws/v1/cluster/scheduler-conf used to update, add, remove queues also update global queue configurations

    • GET
    • PUT

    this API also supports the node label changes
